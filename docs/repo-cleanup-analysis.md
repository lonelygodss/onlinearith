# Onlinearith cleanup analysis and migration plan

## Goal

Clean the `onlinearith` repository so Codex can work from a coherent, low-noise context before implementing the Qwen3-8B OOM fixes. This document is intentionally about cleanup and migration only. The exact MX baseline chunking fix and the lite/full stats fix should happen in a later task.

The cleaned repository should preserve the existing PPL methodology while making these facts obvious:

- `onlinearith` is the experiment driver repo.
- The modified Qwen3 implementation lives in the sibling `transformers` repo.
- `experiment_config.py` is the source of truth for setup IDs and config overrides.
- `modeling_qwen3.py` is the current operational Qwen3 implementation unless a future task explicitly regenerates from `modular_qwen3.py`.
- Deep-pipeline material is abandoned or archived, not part of the immediate path to the OOM fixes.

## Current-state problems

### 1. Agent guidance is stale and too compressed

`CLAUDE.md` is written as one long compressed note. It contains useful facts, but it is hard for Codex to scan and it targets Claude Code rather than Codex. It also encodes old assumptions such as Qwen3-0.6B as the main target and says deep pipeline should be ignored. That should become explicit, structured Codex guidance in `AGENTS.md`.

### 2. Source-of-truth ambiguity between `modular_qwen3.py` and `modeling_qwen3.py`

The README says `modular_qwen3.py` is the main implementation and `modeling_qwen3.py` is autogenerated. The current Claude guidance and the `modeling_qwen3.py` header say the opposite operationally: `modeling_qwen3.py` has been manually extended and is production-relevant. This ambiguity is dangerous for Codex because it may patch the wrong file or regenerate away manual changes.

Cleanup decision: document `modeling_qwen3.py` as the current authoritative implementation for OOM-related work. Move converter details into a developer note and treat modular regeneration as a separate explicit workflow.

### 3. Root directory mixes source, old notes, generated outputs, and experiments

The repo root currently contains active scripts, old plan files, paper notes, result JSON, `__pycache__`, visualization code, quick experiments, and calibration docs. This makes it hard for an agent to determine what is active.

Cleanup decision: keep active CLI entry points at root for compatibility, but move non-entry documentation into `docs/` and remove generated artifacts from git tracking.

### 4. `.gitignore` is too small

The current `.gitignore` only ignores a few calibration/chart folders. It does not cover Python caches, PyTorch artifacts, result JSON files, logs, local data directories, generated benchmark files, or model/checkpoint folders.

Cleanup decision: expand `.gitignore` before moving files so generated outputs do not keep reappearing in future Codex sessions.

### 5. Generated results are tracked

`ppl_batch_summary.json` is a generated output from a past run. Keeping it in root makes Codex treat it as part of the source state and may cause accidental edits or stale conclusions.

Cleanup decision: move historical results to an ignored archive outside source control, or remove them from git tracking after confirming they are backed up under `../data` or a lab notebook.

### 6. Tests and quick experiments contain hardcoded local paths

`test_mxfp8linear.py` inserts `/home/xzjnew/coding/transformers/src` into `sys.path`. This is fragile and will mislead Codex on another machine. Some quick scripts are more like scratch experiments than tests.

Cleanup decision: use either `PYTHONPATH=../transformers/src` in commands or a small helper that resolves the sibling transformers path relative to the repo. Avoid personal absolute paths.

### 7. PPL code is duplicated across `ppltest.py` and `ppl_batch.py`

Both scripts implement sliding-window PPL logic. This duplication is manageable but risky before OOM fixes, because later changes like `use_cache=False`, `logits_to_keep`, and exact label slicing must be applied consistently.

Cleanup decision: do not refactor this first. Mark as a phase-2 cleanup after smoke tests. The first cleanup should only document the duplication and avoid modifying PPL semantics.

### 8. Distributed behavior is not clear enough for 8B

`ppltest.py` and `ppl_batch.py` currently use multiple GPUs as data-parallel workers: each rank loads a full model copy. That improves throughput for small models but does not reduce per-GPU memory. For Qwen3-8B this matters, but changing it is a functional OOM fix, not cleanup.

Cleanup decision: add documentation warning that `--nproc 8` means 8 full replicas in the current scripts. Do not change distributed behavior in cleanup.

### 9. Deep-pipeline material should be quarantined

The active guidance says deep pipeline is abandoned, but setup IDs and text files still mention it. Leaving it in the main mental path increases agent confusion.

Cleanup decision: keep code and setup IDs for backwards compatibility unless the user approves removal, but label deep pipeline as archived/abandoned in README and docs. Move `deep-pipeline.txt` to `docs/archive/deep-pipeline-abandoned.md`.

## Recommended cleanup phases

### Phase 0: Safety snapshot

Before any cleanup, Codex should run:

```bash
git status --short
git branch --show-current
python ppltest.py --list
python calibrate.py --list
```

If there are uncommitted user changes, Codex should not overwrite them. It should either stop and report them or restrict work to new files only.

### Phase 1: Agent and hygiene cleanup

1. Add `AGENTS.md` at repo root.
2. Expand `.gitignore`.
3. Remove `__pycache__/` from git tracking and local tree.
4. Remove or untrack generated result files such as `ppl_batch_summary.json`.
5. Verify active commands still list setups.

This phase should produce no functional Python behavior changes.

### Phase 2: Documentation cleanup

1. Rewrite README into a short active guide.
2. Move long method/baseline/calibration docs under `docs/`.
3. Archive deep-pipeline notes and stale plan files.
4. Preserve links from README to moved docs.
5. Explicitly document that `--nproc` currently creates full model replicas.

This phase should also produce no functional Python behavior changes.

### Phase 3: Test path cleanup

1. Remove hardcoded absolute paths from tests.
2. Import Qwen3 classes from the active implementation path.
3. Prefer `PYTHONPATH=../transformers/src` or a relative bootstrap helper.
4. Run cheap tests.

This is a low-risk code cleanup but should be separate from doc/hygiene commits.

### Phase 4: Optional PPL helper extraction

Only after phases 1-3 pass, consider extracting common PPL helpers from `ppltest.py` and `ppl_batch.py` into a small module such as `ppl_utils.py`:

- `precompute_windows`
- target label construction
- weighted NLL accumulation
- metrics calculation

This is useful before the later OOM fixes because `use_cache=False`, `logits_to_keep`, and label slicing must be applied consistently. However, it touches PPL logic, so it should be guarded by a small equivalence check.

## Cleanup risks

- Moving root scripts without wrappers will break existing commands. Avoid in first pass.
- Regenerating `modeling_qwen3.py` from `modular_qwen3.py` can overwrite manual fixes. Avoid in cleanup.
- README edits can accidentally bless a wrong source of truth. Make it explicit that `experiment_config.py` controls setup IDs and `modeling_qwen3.py` is operational for now.
- Deleting old files before confirming they are non-source can lose useful paper history. Prefer `docs/archive/` over deletion for notes.
- Refactoring PPL before validation can change measured PPL. Defer helper extraction until after hygiene/doc cleanup.

## Handoff to the later OOM-fix task

A clean repo should make the later changes straightforward:

- `experiment_config.py` can add new memory-control config fields such as `mxfp_chunk_target_mib` and stats flags in one place.
- `ppltest.py` and `ppl_batch.py` can be patched consistently for PPL-neutral memory reductions.
- `modeling_qwen3.py` can receive the exact output-chunked MX path with a focused unit test.
- Stats-lite behavior can be tested independently from PPL.

Until then, the cleanup should leave numerical behavior unchanged.
