# Hardware Design Plan

## 1. Scope and design principles

The hardware is organized around two orthogonal ideas:

1. **FFN-deep streaming dataflow with pipelined scatter.**
   `gate_proj` and `up_proj` act as stage-1 producers, `SiLU` and gating remain inside the pipeline, the gated intermediate result crosses an explicit scatter boundary, and `down_proj` stays local to the output shard that owns the destination channels.
2. **Checkpoint-based early termination (ET).**
   Calibrated static budgets gate digit work at projection checkpoints through small local counters. ET decides how many MSD cycles are executed; the deep pipeline and scatter decide how those cycles are transported and overlapped.

The detailed design unit is a representative **P x H FFN micro-tile**:
- `P` owner lanes for intermediate channels `i`
- `H` owned `down_proj` outputs `h`
- MX block size `B = 32`

Representative points such as `P = 8 or 16` and `H = 16 or 32` are sufficient for the paper. The key architectural property is that each tile owns a local `down_proj` output shard, so reduction stays local after the scatter boundary.

## 2. Stream and timing model

The current plan assumes a deterministic source-side schedule.

- For a fixed calibrated budget and fixed compute pipeline, the timing of each owner lane is determined by **static budget length plus fixed path latency**.
- Each owner lane only performs **within-lane alignment** between the gate path and the up path.
- After this local alignment, all owner lanes emit the **first valid mantissa MSD on the same absolute cycle**.
- Different channels differ mainly in the **length of the valid MSD stream**, because calibrated budgets make some channels terminate earlier than others.
- The scatter window is therefore a **common stream start with a decaying tail**, not a first-arrival skew problem.

The stream is split into two coupled payload classes:

1. **Mantissa stream**
   - exact BSD MSD-first digit stream
   - valid/start/end markers
   - source owner ID and destination shard metadata
2. **Exponent/control stream**
   - block exponent and shift metadata
   - route / shard ID
   - control markers for stream start / end
   - exponent contribution generated by the nonlinear path, including the SiLU-side exponent update

The exponent/control path does not depend on the full mantissa MSD sequence and can therefore be launched **no later than** the mantissa stream, and in some cases **earlier**. This allows the destination shard to prepare alignment and delay resolution before the full mantissa traffic has arrived.

## 3. Micro-tile organization

### 3.1 FFN entry and preparation

The FFN entry block performs the format conversion once at the beginning of the FFN pipeline:
- MX decode and block exponent extraction
- mantissa recode into BSD / digit-serial form
- digit broadcast to stage-1 owner lanes
- activation-side metadata latch

The paper should emphasize that conversion happens only at FFN entry and FFN exit, not between internal FFN stages.

### 3.2 Stage-1 owner lanes

Each owner lane `i` contains:
- local `gate_proj` weight bank
- local `up_proj` weight bank
- gate MAC path
- up MAC path
- local stage-1 ET counter loaded from calibrated budget `B1[i]`
- metadata registers for stream start, stream end, and route tags

Stage-1 ET is a static local controller:
- no runtime budget delta
- no combined-scale LUT in the critical path
- no dynamic add-path for budget adjustment

For a given input block, the owner lanes execute in lockstep at the front of the pipeline. The main cross-channel difference is not first-valid timing but **how long each lane remains active**.

### 3.3 Nonlinearity, local alignment, and gating

Per owner lane:
- `PWL SiLU` on the gate path
- within-lane time alignment between `SiLU(g)` and `u`
- gating multiply
- register stage that separates mantissa payload from exponent/control payload

This block is important for the current timing story:
- alignment is only within the lane
- once the fixed pipeline depth is paid, the first valid mantissa digit appears at the same cycle for all lanes
- calibrated budgets shorten or extend the **tail** of the gated stream rather than moving its first valid cycle

The exponent contribution produced by the SiLU/gating path should be carried on the exponent/control path rather than staying embedded in the mantissa stream.

### 3.4 Pipelined scatter boundary

The scatter boundary sits between the gated owner streams and the local `down_proj` consumer bank.

It contains:
- mantissa stream packetization or flitization
- separate exponent/control routing path
- destination-shard selection
- per-destination elastic FIFOs / credits
- issue arbitration

Pipelined scatter has the following semantics:
- scatter operates on the **stream as it is produced**, not after materializing a full intermediate vector
- all channels can begin scattering at the same first-valid cycle
- the number of active mantissa streams then decreases over time as shorter-budget channels terminate
- exponent/control traffic can arrive earlier and complete earlier than the mantissa traffic

This allows a narrower interconnect than a barriered whole-slice transfer and reduces stage-boundary burstiness, while still overlapping transport with the late tail of the mantissa window.

### 3.5 Stage-2 local `down_proj` consumer bank

Each tile owns a local bank for `H` output channels.

It contains:
- local `down_proj` weight bank
- ingress FIFOs from scatter
- exponent/control receive path for shift and delay resolution
- local mantissa consumer datapath
- local accumulators / reduction tree for the owned outputs
- optional stage-2 ET storage `B2[h]`

The current base plan keeps stage-2 simple:
- exponent/control arrival is used to start **delay resolution, shift selection, and local setup early**
- the mantissa-side `down_proj` operation is still treated as a dot-product style local reduction
- the hardware plan does **not** rely on aggressive speculative partial accumulation before the required mantissa window is available, because the added control complexity is high and the expected gain is modest

In other words, early overlap is mainly harvested on the **transport and control-preparation side**, while the mantissa reduction remains a disciplined local compute stage.

### 3.6 Local metadata and control SRAM

A small side-band control block stores:
- `B1[i]` SRAM for stage-1 calibrated budgets
- optional `B2[h]` SRAM for stage-2 calibrated budgets
- weight exponent SRAM
- route / shard-ID metadata
- short counters and active flags
- coefficient ROM for the PWL SiLU block

This keeps the hardware story concrete without introducing dynamic prediction logic.

## 4. Early termination implementation

### 4.1 Stage-1 ET controller

For each owner lane:
- load `B1[i]` at stream start
- decrement once per issued valid MSD digit
- when the counter expires, gate:
  - weight SRAM read
  - digit MAC toggle
  - nonlinear / multiply toggle
  - mantissa scatter write
  - end-of-stream metadata generation

This is a local down-counter implementation, not a runtime significance predictor.

### 4.2 Stage-2 ET policy

Stage-2 ET can reuse the same static primitive if it materially improves the Pareto frontier. For the mainline hardware story, it should remain secondary to stage-1 ET and should not complicate the `down_proj` timing narrative.

### 4.3 Optional exact-zero side path

If exact-zero information is already available from quantization or encoding, it can be exposed as a small bypass or null-stream marker. It should remain a minor side path rather than the main control mechanism.

## 5. Timeline that the figures should show

The timing figure should show a **same-start, different-length** stream pattern.

Suggested visual sequence:
- `gate_proj` and `up_proj` execute in lockstep under `B1[i]`
- fixed SiLU and align latency is paid once
- all owner lanes emit their first mantissa MSD at cycle `t0`
- exponent/control traffic is issued at `t0` or earlier if available
- active mantissa stream count decays over the next tens of cycles as channels finish
- destination shards receive exponent/control information early enough to resolve local setup
- mantissa-side `down_proj` reduction starts under the simpler local policy rather than a speculative partial-dot-product schedule

A good subfigure is an active-stream-count trace:
- cycle on the x-axis
- number of live owner streams on the y-axis
- separate line for exponent/control traffic completion if helpful

## 7. Minimal quantitative annotations for the hardware section

Illustrative values are enough:
- `B1[i]`, `B2[h]`: 6 bits
- ET counter: 6 bits
- route / shard ID: 3 to 5 bits
- source-owner ID: `log2(P_total)` bits
- align FIFO depth: 4 to 8 entries
- scatter FIFO depth: 4 to 8 entries
- SiLU segment index: 3 bits for an 8-segment PWL block

These are not hard commitments, but they make the macro feel implementable.

## 8. Final hardware message for the paper

The hardware section should land on the following interpretation:

> The FFN macro uses calibrated static checkpoint ET to bound digit work, while an FFN-deep streaming pipeline transports exact BSD mantissa streams and earlier exponent/control metadata across a pipelined scatter boundary. All owner lanes share the same first-valid cycle after local alignment, but their calibrated stream lengths differ, creating a decaying scatter window that reduces burst bandwidth and allows early consumer-side setup without requiring a complicated speculative `down_proj` reduction schedule.
