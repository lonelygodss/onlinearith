     # MSD-First Time-Domain Truncated Dot-Product Simulation: Project Summary

     ## 1. How is it Constructed

     ### 1.1 Architecture Overview

     This is a software simulation of a custom Compute-in-Memory (CiM) dot-product hardware unit for LLM
     inference, implementing **MSD-first (Most Significant Digit) digit-pipelined arithmetic with BSD (Binary
     Signed-Digit) representation**. The simulation is built as modifications to the Qwen3-0.6B model in
     HuggingFace Transformers.

     ### 1.2 Core Components

     **Modified Transformers Source** (at `../transformers/src/transformers/models/qwen3/`):
     - `modeling_qwen3.py` — Main implementation with custom MXFP linear layers and MSD truncation logic
     - `configuration_qwen3.py` — Extended config supporting MXFP/MSD parameters
     - `calibration_msd.py` — Offline budget calibration engine with binary search and fixed-sum optimization
     - `msd_perf_stats.py` — Hierarchical performance statistics collection (utilization, sparsity, cycle counts)

     **Evaluation Scripts** (this directory):
     - `experiment_config.py` — Central configuration hub defining all 21 setups and baseline parameters
     - `ppltest.py` — Single-setup perplexity evaluation with window-level data parallelism
     - `ppl_batch.py` — Batch evaluation across all setups with setup-level parallelism
     - `calibrate.py` — Multi-format MSD budget calibration wrapper
     - `calibrate_base.py` — Baseline N:M structured sparsity calibration
     - `dist_utils.py` — Distributed training utilities (NCCL, auto-relaunch, port management)

     ### 1.3 Three-Tier Budget Resolution System

     The MSD truncation system uses a hierarchical budget composition:

     **Tier A — Uniform Budget**: All channels use global `msd_cycle_budget` (default: 16 cycles)

     **Tier B — Calibrated Budget**: Per-channel base budgets from offline calibration
     - **SNR-min mode**: Binary search finds minimum budget meeting target SNR (30 dB default)
     - **Fixed-sum mode**: Redistributes cycles across channels to minimize total error while preserving hardware
     budget

     **Tier C — Dynamic Budget**: Runtime adjustment based on activation+weight scale exponents
     ```
     B_final[n,j] = B_base[j] + ΔB_dynamic[n,j]
     ```

     ### 1.4 Quantization Formats

     Four mutually-exclusive MXFP (microscaling) block quantization formats (block_size=32):
     - **MXFP8**: E4M3 (max=448.0)
     - **MXFP6 E2M3**: max=7.5
     - **MXFP6 E3M2**: max=28.0
     - **MXFP4**: E2M1 (max=6.0)

     ### 1.5 Deep Pipeline Architecture

     Simulates explicit MSD digit timing flow through FFN stages using **float carrier + timing metadata**:

     ```python
     @dataclass
     class MSDFlowTensor:
         value: torch.Tensor        # (N, C) float32 numeric value
         first_cycle: torch.Tensor  # (N, C) first MSD digit arrival time
     ```

     **FFN Pipeline Flow**:
     ```
     Input → [Prepare] → gate_proj ⊕ up_proj (producer mode)
                              ↓           ↓
                         MSDFlowTensor  MSDFlowTensor
                              ↓
                         [PWL SiLU] → MSDFlowTensor
                              ↓           ↓
                         [Gating Multiply] → MSDFlowTensor
                                   ↓
                              down_proj (consumer mode)
                                   ↓
                               Output
     ```

     **Two-Domain Budget Ownership**:
     - **Stage-1 shared budget** `B_stage1[n,i]`: governs gate_proj, SiLU, gating multiply
     - **Stage-2 consumer budget** `B_down[n,h]`: governs down_proj only

     **PWL SiLU**: 8-segment piecewise-linear approximation with 5-cycle fixed latency (3-cycle MAC + 2-cycle
     multiply)

     ## 2. What Behavior Can It Simulate

     ### 2.1 Arithmetic Support

     **(1) MXFP4/6/8 Quantization**
     - Standard microscaling block quantization (block_size=32)
     - Shared exponent per block, individual mantissas

     **(2) FFN Layer Deep Pipeline MSD-first Online Arithmetic**
     - MSD metadata passed between FFN stages (gate_proj → SiLU → multiply → down_proj)
     - Explicit timing propagation with `first_cycle` metadata
     - Accuracy and cycle dependence simulated through pipeline
     - Channel cycle budget generated by calibration

     ### 2.2 Sparsification Support

     **(1) Baseline Structured Sparsity (Wanda-style N:M)**
     - Weights are MXFP-quantized **before** computing `|W_ij|` to account for format truncation
     - Structured N:M sparsity based on layer L2 norm optimization: N elements pruned per M-element group (per output row)
     - Offline calibration generates boolean masks, weights zeroed before inference
     - Inference runs at native speed with standard MXFP calculations

     **(2) SNR-Gated Cycle Budget (Per-Channel)**
     - Binary search finds minimum per-channel budget meeting target SNR (30 dB)
     - Early termination when budget exhausted

     **(3) Fixed-Sum Optimized Cycle Budget**
     - Redistributes cycles from low-gain to high-gain channels
     - Preserves total hardware budget while minimizing layer error
     - Greedy heap-based swap algorithm exploiting asymmetric error sensitivity

     **(4) Dynamic Budget Adjustment**
     - Runtime adjustment based on activation+weight scale exponents
     - Controlled by `msd_budget_dynamic_scale` and `msd_budget_dynamic_threshold`
     - Modes: `linear`, `exponential`, `step`
     - Always added on top of base budget (Tier C)

     **(5) Deep Pipeline Early Termination**
     - Stage-1 shared endpoint budget for gate_proj/up_proj/SiLU/multiply
     - Stage-2 consumer budget for down_proj with upstream timing propagation

     **Performance Statistics**:

     **(1) MAC Sparsity Metrics**:
     - **Utilization**: Fraction of non-zero MACs executed
       ```
       utilization = active_macs / total_macs
       ```
     - **Zero Block Fraction**: Fraction of blocks totally truncated
       ```
       zero_block_frac = zero_blocks / total_blocks
       ```
       (Idle blocks can be power-gated with lower cost than partial activation)

     **(2) Cycle Statistics**:
     - Per-layer cycle counts: `{layer_name: total_cycles}`
     - Per-projection breakdown: gate_proj, up_proj, down_proj
     - Histogram of cycle budgets used


     ## 4. Highlights of This Simulation

     ### 4.1 Hierarchical Budget System

     **Three-tier composition** enables flexible exploration:
     - Tier A (uniform) for quick prototyping
     - Tier B (calibrated) for hardware-realistic scenarios
     - Tier C (dynamic) for adaptive runtime optimization

     This mirrors real hardware design space: fixed budgets (simple control), calibrated budgets (offline
     optimization), dynamic budgets (runtime adaptation).

     ### 4.2 Accurate Timing Simulation

     Unlike scalar-budget proxies, we uses **explicit timing metadata** (`first_cycle`) to
     track when MSD digits arrive at each stage. This enables:
     - Accurate cross-stage dependency modeling
     - Realistic pipeline stall simulation
     - Precise cycle accounting for complex dataflow (SiLU, gating multiply)

     ### 4.3 Unified MXFP + Sparsity Framework

     The simulation integrates two orthogonal optimization dimensions:
     - **Quantization** (MXFP4/6/8): reduces precision
     - **Sparsification** (N:M structured, MSD truncation): reduces computation

     Both are calibrated offline using the same infrastructure, enabling fair comparison
     and combined optimization.

     ### 4.4 Hardware-Realistic Constraints

     **Cycle budgets**: directly map to hardware clock cycles, it's a hardware firendly unstrtuctured sparsity

     **Deep pipeline**: models real pipelined MAC array with stage-to-stage data dependencies, it's a transaction level timing model

     This enables realistic assessment of hardware design choices on real LLM workloads, not toy problems.
